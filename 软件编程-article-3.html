用了 7 年 Web 组件，踩坑无数，我终于看清了的真相

📌 一句话摘要

作者作为框架作者,结合七年实践经验,深入分析了 Web 组件的局限性与风险,认为其可能成为 Web 发展的陷阱,阻碍 Web 创新与发展。

📝 详细摘要

本文作者作为 SolidJS 框架作者,基于七年的 Web 组件生产实践经验,对 Web 组件的适用性提出了质疑。文章剖析了 Web 组件试图通过标准化实现跨框架互操作的乌托邦愿景,但同时也指出这种标准化可能带来的创新阻碍。作者认为,Web 组件的抽象方式增加了复杂性,导致性能开销和开发成本的增加。此外,Web 组件对 Shadow DOM 的依赖也限制了 SSR 和水合等技术的优化,例如事件冒泡问题。作者最终得出结论,Web 组件虽然在某些场景下易于使用,但其带来的代价是巨大的,甚至可能对 Web 的未来发展构成潜在风险。因此,作者更倾向于使用成熟的框架,因为它们更可控、更高效、更易于优化,而非依赖 Web 组件来实现跨框架的组件共享。

💡 主要观点

  1. Web 组件的标准化可能限制创新 过早的标准化会固化某些技术选择,限制开发者探索替代方案,从而阻碍 Web 技术的创新与发展。
  2. Web 组件的抽象增加了复杂性和开销 Web 组件基于自定义元素,需要通过 DOM 进行交互,增加了额外的复杂性,导致性能开销和开发成本的增加。
  3. Web 组件并非解决所有问题的银弹 Web 组件在隔离性和可移植性方面有优势,但在微前端和组件共享等场景下,其优势并不明显,且存在额外的开销。

💬 文章金句

  • Web 组件可能是 Web 未来面临的最大潜在风险。
  • 真正危险的是,它们被赋予了自身并不具备的特性,做出了无法兑现的承诺。正是它们的存在扭曲了周围的一切,使整个 Web 处于危险之中。
  • Web 组件是一种彻头彻尾的妥协。有时我们确实需要做出妥协。但这没什么值得兴奋的。
  • Web 组件的愿景是:终有一天,开发者无论使用何种工具编写组件,都能像使用原生 DOM 元素一样,将其无缝集成到任何网站,而无需考虑网站的构建方式。
  • 当原语超出了预期的使用范围,或者说过度抽象时,你就无法挽回了。
