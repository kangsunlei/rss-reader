Visual Studio 包加载机制更新：避免循环依赖

2/13/2025, 10:45:37 PM

📌 一句话摘要

Visual Studio 更新包加载机制，避免循环依赖导致的竞争条件，建议开发者精简 Initialize/SetSite 方法中的操作。

📝 详细摘要

本文讨论了 Visual Studio 包加载机制的更改，解决了由异步加载和现有同步包加载引起的竞争条件问题。新的限制避免了循环包加载请求，从而防止服务请求返回空值。此更改在 17.12 预览 2 中实现，增强了 Visual Studio 的稳定性和一致性。开发人员需要调整代码以避免循环依赖，例如延迟服务检索和工具窗口创建，使用 Lazy 或 AsyncLazy。这旨在减少空服务出现的概率，确保更可预测的行为。

💡 主要观点

  1. Visual Studio 调整包加载机制，禁止循环依赖，以避免因竞争条件导致服务返回空值。 此改进通过避免服务请求意外返回空值的场景，提高了系统稳定性。
  2. Initialize(Async)/SetSite 方法应尽可能减少操作。 避免在 Initialize(Async)/SetSite 中急于创建服务实例或工具窗口，以防止循环依赖。
  3. 使用 Lazy 或 AsyncLazy 实现服务“即时 (Just-In-Time)”检索。 在 Initialize(Async)/SetSite 之外，使用 Lazy 或 AsyncLazy 可以实现服务的“即时”检索，从而降低循环依赖的风险。
  4. AsyncLazy 负责管理包加载，确保只有在 Initialize(Async)/SetSite 完成后，包才被视为已加载。 通过确保在包完全加载前注册服务工厂，从而避免竞争条件。

💬 文章金句

  • 过去不建议使用循环包加载。之前的成功仅仅是因为 25 年前 Visual Studio 主要为单线程，且包加载必须在 UI 线程上进行。
  • 您的 Initialize(Async)/SetSite 方法应该尽可能减少操作。
  • 为避免循环依赖，可以在 Initialize(Async)/SetSite 之外，使用 Lazy 或 AsyncLazy 实现服务的“即时 (Just-In-Time)”检索。
