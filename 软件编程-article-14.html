淘宝订单列表 Fragment 转场动画卡顿问题及 SurfaceView + 快照解决方案。

📝 详细摘要

本文深入分析了淘宝订单列表 Fragment 转场动画卡顿的原因,主要是由于 Fragment 动画在主线程执行,容易被 UI 绘制阻塞。文章对比了 Activity 和 Fragment 动画的差异,提出了三种解决方案:动画延后、分帧渲染和异步转场动画框架 (SurfaceView + 快照)。动画延后虽然简单但会增加点击响应耗时;分帧渲染通过 Choreographer.FrameCallback 将渲染任务拆分,但仍无法完全避免丢帧;最终,采用了 SurfaceView + 快照的方案,利用 SurfaceView 的独立渲染能力,通过异步渲染解决了卡顿问题,实现了性能提升和用户体验改善,同时兼顾了点击响应耗时和流畅度。该方案实施中的一个关键技术点是使用 PixelCopy.request() 方法获取快照。

💡 主要观点

  1. Fragment 转场动画卡顿是由于主线程阻塞导致。 Fragment 动画在主线程执行,当主线程被 UI 绘制等耗时任务阻塞时,会导致动画卡顿,影响用户体验。
  2. SurfaceView 具备独立渲染能力,可用于解决动画卡顿问题。 SurfaceView 拥有独立的 Surface,可以在单独的线程中进行绘制,避免占用主线程资源,从而提升渲染性能。
  3. SurfaceView + 快照方案通过异步渲染解决了卡顿问题,同时优化了点击响应耗时。 该方案利用 SurfaceView 异步执行动画,Fragment 渲染完成后隐藏 SurfaceView,兼顾了点击响应耗时和动画流畅度。

💬 文章金句

  • Activity 转场动画不会被主线程阻塞,而 Fragment 动画则会被主线程任务阻塞。
  • SurfaceView 能够独立参与窗口管理,享有与 DecorView 类似的管理机制,并且它被允许在不干扰 UI 线程的情况下,由专有线程高效地进行图形渲染。
  • 在最大程度缩减点击响应耗时的同时保证转场动画丝滑流畅,并且在动画期间不存在 loading 状态,这对于用户体验来说是巨大的提升。
