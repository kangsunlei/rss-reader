得物 Android 团队分享了解决 Android 平台上多个复杂崩溃问题的实践，涵盖 DNS 解析、MediaCodec、多线程环境及 Android 15 等，展示了深入的问题分析和创新性的解决方案。

📝 详细摘要

本文总结了得物 Android 团队在 Crash 治理方面的实践经验。文章概述了通过修复历史遗留的 Crash 漏报问题，得物 Android 端的 Crash 监控体系得到显著增强，崩溃率从最高的万分之二降至目前的万分之 1.1 到万分之 1.5。接着，详细介绍了四个典型的系统 Crash 治理过程：DNS 解析崩溃（采用 inline-hook 和信号处理机制）、MediaCodec 状态异常崩溃（利用 C++ 异常处理机制）、bio 多线程环境崩溃（实施 JNI hook 和读写锁）以及小米 Android 15 焦点处理空指针崩溃。针对每个问题，文章都深入分析了崩溃原因，并提供了相应的解决方案。这些实践经验对于其他 Android 开发者在进行 Crash 治理时具有重要的参考价值，展示了得物团队在解决复杂问题上的独特思路。

💡 主要观点

  1. 通过 inline-hook 代理 android_getaddrinfofornet 函数，并结合信号处理机制，将 Native Crash 转化为 Java 异常，解决了 DNS 解析崩溃问题。 捕获调用原函数过程中出现的段错误信号，并将其转换为 JAVA 异常，最终走进 parseNumericAddressNoThrow 方法的容错逻辑。
  2. 通过 hook Android 11 并使用 C++ 的 try-catch 异常处理机制，对齐 MediaCodec 在 Android 11 和 12 上的逻辑，解决了状态异常崩溃，避免了栈溢出问题。 使用 C++ 的 try catch 机制来做跨函数域的跳转，解决了栈溢出保护导致错误转移的问题。
  3. 通过 JNI hook 代理关键函数，并在 Native 层实现与 Android 12 相同的读写锁逻辑，解决了 bio 多线程环境下的野指针崩溃问题。 在 Native 层实现与 Android 12 相同的读写锁逻辑，避免了多个线程同时 close 连接时出现的野指针问题。

💬 文章金句

  • Android11 及以下版本在 DNS 解析过程中的有几率产生野指针问题导致的 Native Crash，其中 Android 9 占比最高。
  • 解决问题的关键在于深入理解系统源码，并找到合适的 hook 点和异常处理机制，实现逻辑对齐和错误兜底。
  • Android 11 Socket close 过程中在多线程场景下有几率产生野指针问题导致 Native Crash。
