快手 Java 透明协程：实现零代码修改提升 30%QPS

2/12/2025, 8:17:30 AM

📌 一句话摘要

快手自研 Java 17 透明协程技术，在对业务无侵入的前提下，解决了社区版本调度、抢占和 IO 管理的缺陷，实现服务 QPS 提升 30%，并显著降低成本。

📝 详细摘要

文章详细介绍了快手基于 Dragonwell 社区 Wisp 协程自研的 Java 17 **透明协程**技术。针对传统线程模型和异步模型的局限性，快手选择了协程这一“同步编程，异步执行”的方案。文章深入分析了快手在协程落地过程中遇到的挑战，包括低负载下 CPU 消耗高、长任务抢占开销大以及 IO 查询不及时等问题。针对这些问题，快手团队对调度器、抢占机制和 IO 模型进行了深度优化和重构。通过采用 Handshake 和 HandOff 机制降低抢占开销，并优化 IO 管理模块，解决了 JNI 长任务抢占和 IO 查询不及时的问题。最终，快手 Java 协程架构在 Java 17 上得以实现，服务 QPS 提升 30% 以上，并节省了数千万的服务器成本。**该方案对业务具有非侵入性，是其一个重要优势。**

💡 主要观点

  1. 快手通过优化调度器，有效控制了低负载下协程的 CPU 消耗 通过最小化线程数和连续执行原则，减少 Context-Switch，消除了低负载工况下相对于传统线程模型的 CPU 效率劣势。
  2. Handshake 和 HandOff 机制显著降低 Java 和 JNI 长任务抢占开销 Handshake 机制避免了 StopTheWorld，降低 Java 长任务抢占代价；HandOff 机制实现了 JNI 长任务的及时抢占，提升了抢占性能。
  3. IO 模型优化显著提升了 IO 查询的响应速度，缩小了与线程模型的性能差距，并在某些场景下实现了超越 通过非阻塞 IO 补偿、复用内核结构和资源缓存限定等手段，显著提升了 IO 查询的响应速度，缩小了与线程模型的性能差距。

💬 文章金句

  • 协程以 '同步编程，异步执行' 平衡两者，成为现代语言标配。
  • 协程的主要优势在于减少了内核线程的上下文切换指令开销。更为关键的收益在于协程显著改善了内核 CFS 的调度延迟。
  • 抢占的本质目的在于消除长任务执行对其它任务的影响，虽然 JNI 没有类似 Handshake 的机制能够打断特定任务，但如果将受影响的任务及时转交给其它的 WispCarrier 进行补偿，同样也可以消除长任务的影响。
