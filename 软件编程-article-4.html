Bun 迎来重要更新：新增 Chrome 调试、S3 API，引发架构争议

2/9/2025, 1:09:33 AM

📌 一句话摘要

Bun 1.1.3 更新引入 S3 API 等功能,但其架构设计引发争议。

📝 详细摘要

Bun 发布了 1.1.3 版本,该版本引入了 S3 API、Chrome 调试支持等新功能,并修复了大量问题。S3 API 允许开发者直接在 Bun 中使用 AWS S3 服务,Chrome 调试支持则方便开发者调试 Bun 内存使用情况。同时,Bun 还改进了 Node.js 的兼容性,并计划在 1.2 版本中推出 PostgreSQL 客户端。Bun 使用 Zig 语言编写,并使用 WebKit 的 JavaScriptCore 作为其 JavaScript 引擎,从而减少启动时间和内存使用量。然而,将 S3 API 等特定于供应商的 API 集成到 Bun 核心中是否合适引发了一些争议,有开发者认为这些功能更适合放在库中,以避免造成 Bun 的臃肿和维护负担。

💡 主要观点

  1. Bun 1.1.3 引入 S3 API 和 Chrome 调试支持,提升开发效率 S3 API 方便开发者直接使用 AWS S3 服务,简化与 AWS S3 的集成,Chrome 调试支持则方便开发者调试 Bun 内存使用情况,从而提升开发效率。
  2. Bun 持续改进 Node.js 兼容性,但仍未完全替代 Bun 在 Node.js 兼容性方面取得了进展,但仍然无法完全替代 Node.js,兼容性改进占据了团队大部分时间。
  3. S3 API 集成引发争议,或造成 Bun 长期臃肿 将 S3 API 等特定于供应商的 API 集成到 Bun 核心中是否合适引发了一些争议,有开发者认为这些功能更适合放在库中。

💬 文章金句

  • 在运行时中使用特定于供应商的 API 不是一个好主意
  • S3 已经超过 15 岁了。有足够多的旧 S3 客户端和与兼容 S3 的 API,破坏 API 的更改实际上是不可能发布的。
  • 即使 if 语句没有嵌套,if 语句的语句解析步骤也始终是递归的。这种递归占用了太多的堆栈空间。现在它不再是递归的了。
  • 喜欢其内置电池的态度
  • 与谷歌 Chrome 和开源 Chromium 项目所使用的更广泛的 V8 引擎相比,这减少了启动时间和内存使用量。
