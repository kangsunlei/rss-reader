📌 一句话摘要

文章综述了近期在语音交互、文本生成以及 OpenAI GPT-4.5 和 Anthropic Claude 3.7 Sonnet 等 AI (人工智能) 模型方面的最新进展。

📝 详细摘要

本期 deeplearning.ai Batch 探讨了语音交互中 VAD (语音活动检测) 的挑战，并介绍了 Kyutai Labs 的 Moshi 模型如何通过持续监听解决这一问题。文章还介绍了 Inception Labs 的文本生成扩散模型 Mercury Coder，强调其基于扩散模型和速度快的特点。同时，文章对比分析了 OpenAI 的 GPT-4.5，强调其规模庞大但并非推理模型，以及 Anthropic 的 Claude 3.7 Sonnet，突出其混合推理方法和用户可控的推理 tokens 数量。此外，文章还提到了 OpenAI 面临的 GPU (图形处理器) 短缺问题，以及 Anthropic 的 Claude Code 工具。

💡 主要观点

  1. 语音交互中的 VAD (语音活动检测) 在噪声环境下表现不佳，Moshi 模型通过持续监听解决此问题。 传统的语音活动检测 (VAD) 在嘈杂环境中难以准确判断用户何时说话，Moshi 模型通过持续的双向音频流，让模型自行判断何时响应，无需显式的 VAD 步骤。
  2. Inception Labs 的 Mercury Coder 使用扩散模型进行文本生成，速度较快。 Mercury Coder 通过扩散模型逐步去除噪声来生成代码，在速度上优于传统的自回归模型，但在某些 benchmark 上不如 DeepSeek Coder V2 Lite。
  3. OpenAI 发布 GPT-4.5，强调规模而非推理能力，但在 PersonQA 上准确率有所提升，但面临 GPU (图形处理器) 短缺。 GPT-4.5 是一个大型通用知识模型，虽然在某些方面有所改进，但并非推理模型。在 PersonQA 测试中，GPT-4.5 相比 GPT-4o 准确率有显著提升，但由于 OpenAI 面临 GPU (图形处理器) 短缺，其 API (应用程序编程接口) 可用性受到影响。
  4. Anthropic 的 Claude 3.7 Sonnet 采用混合推理方法，允许用户控制推理的 tokens 数量。 Claude 3.7 Sonnet 在编码和前端 Web 开发方面表现出色，支持工具使用和计算机使用，并允许用户在即时响应和扩展思考模式之间切换，通过控制推理 tokens 数量来优化输出。
  5. 语音交互领域仍有巨大的创新潜力，解决延迟和 VAD (语音活动检测) 错误等技术问题将推动语音应用的发展。 尽管语音交互技术取得了显著进展，但延迟和 VAD (语音活动检测) 错误等问题仍然存在。解决这些技术难题将为语音交互应用开辟更广阔的前景。
