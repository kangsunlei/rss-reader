本文深入分析了 Pnpm 包管理器的核心优势，对比 npm 和 Yarn，重点探讨了其在解决幽灵依赖和优化依赖治理方面的创新方案，为前端项目提供高效依赖管理指导。

📝 详细摘要

文章详细剖析了前端包管理工具的发展历程，从 npm 到 Yarn，再到 Pnpm，着重讲解了 Pnpm 如何通过独特的**硬链接**和**软链接**存储机制解决传统包管理器的依赖冗余、安装效率低等问题。文章深入探讨了幽灵依赖的产生原因和 Pnpm 的应对策略，并提供了 Monorepo 项目中依赖治理的实践方案，包括识别冗余依赖和重叠依赖。通过对比 npm、Yarn 和 Pnpm 的安装流程和依赖管理方式，突出了 Pnpm 在安装速度和磁盘空间利用上的性能优势，尤其适用于大型项目和 Monorepo 架构。最后，文章也指出了 Pnpm 生态仍在发展中的现状，但仍然推荐拥抱 Pnpm，以提升前端工程效率。

💡 主要观点

  1. Pnpm 通过内容寻址存储和硬链接，避免了重复安装，节省磁盘空间 Pnpm 将依赖存储在全局 store 中，通过硬链接在不同项目间共享，避免了 npm 和 Yarn 中依赖重复安装的问题，极大地节省了磁盘空间。
  2. Pnpm 使用软链接创建严格的依赖结构，有效解决了幽灵依赖问题 Pnpm 通过软链接将依赖连接到 .pnpm 目录下的真实位置，使得项目只能访问 package.json 中声明的依赖，避免了访问未声明依赖的风险。
  3. Pnpm 对直接依赖严格管理，对间接依赖妥协处理 考虑到一些第三方库可能依赖于未直接声明的包（幽灵依赖），Pnpm 默认启用了 hoist 配置，在保持依赖隔离的同时，允许某些特殊情况下的间接依赖被访问，实现了依赖管理上的平衡。
  4. Monorepo 项目可以通过脚本识别重叠依赖，并制定合理的依赖管理策略 文章提供了一个脚本示例，用于识别 Monorepo 项目中根目录和子项目的重叠依赖，并提出了将共享依赖提升至根目录、使用 resolutions 等依赖管理策略。

💬 文章金句

  • pnpm 下载的依赖全部都存储到 `store` 中去了，`store` 是 pnpm 在硬盘上的公共存储空间。
  • Pnpm 通过使用全局的 `.pnpm-store` 来存储下载的包，使用硬链接来重用存储在全局存储中的包文件，这样不同项目中相同的包无需重复下载，节约磁盘空间。
  • 依赖处理方式：依赖包 ---(软链接)---> .pnpm ----(硬链接) ---> 全局的 Store
  • pnpm 把会包下载到一个公共目录，如果某个依赖在 `sotre` 目录中存在了话，那么就会直接从 store 目录里面去 `hard-link`，避免了二次安装带来的时间消耗
