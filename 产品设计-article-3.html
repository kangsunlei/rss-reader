重写了 6 年前的 PRD，发现 4 个需求文档深坑，新人一踩一个准，产品经理避雷速看

2/26/2025, 3:02:31 PM

📌 一句话摘要

本文总结了产品经理新人编写音频类知识付费 App 的 PRD 时常见的四大问题，并提供了实用的避坑指南。

📝 详细摘要

本文作者以自身经历为引，回顾了 6 年前编写的音频类知识付费 App 的需求文档，指出了新人产品经理在 PRD 编写中常见的四大问题。这些问题包括：过度关注 UI 样式和交互细节，导致描述冗余；忽略前端数据在后端的取值来源及逻辑；缺乏对常规业务逻辑规则的说明，容易遗漏边界情况；以及只关注 App 端，忽略后台管理功能的重要性。作者针对这些问题，给出了具体的修改示例，强调 PRD 应关注数据来源、逻辑完整性以及后台功能，并建议产品经理关注核心功能和 MVP 原则。

💡 主要观点

  1. PRD 编写应避免过度描述 UI 样式和交互细节 对于常规交互，只需正常描述；UI 设计介入后，PRD 可省略样式描述，除非没有 UI 设计时才需要补充。
  2. PRD 需明确数据取值来源，避免仅关注前端展示 页面展示信息通常来自接口调用，PRD 应清晰标明数据的接口来源及对应字段。
  3. PRD 需包含常规逻辑规则说明，考虑各种状态和边界情况 PRD 容易遗漏排序、极值、状态处理等逻辑，应尽可能考虑全面，并不断积累经验。
  4. PRD 编写需涵盖后台功能需求 任何用户端产品都需要后台管理系统，PRD 应涵盖后台功能需求，如内容审核、用户管理等。

💬 文章金句

  • 产品经理永远不可能全部考虑到，研发永远不可能写不出 bug，测试永远不可能覆盖全部测试点。
  • PRD 写的不行我认，认了我就改，改了下次我就不再犯。
  • 当挨怼挨的多了，慢慢的就写的好了。
  • 当定位成 MVP，也就是最小可行产品，就是尽量以最小的成本和最短的时间推出具备基本功能的产品，以验证产品的可行性、吸引用户并收集反馈。
